Upgrade AskiMate chat from polling-based pseudo-live updates to proper live real-time updates with smooth UX on BOTH user and admin sides.

This phase is ONLY about:

==================================================
REQUIRED CHANGES
==================================================

1. Replace polling-based message refresh for AskiMate chat with real-time live updates.
   Preferred approach:
   - use WebSocket if the stack supports it cleanly
   - otherwise use Server-Sent Events as fallback
   - do NOT use periodic full refetch polling for active chat windows anymore

2. Live updates must work on BOTH:
   - user chat page
   - admin conversation page

3. When a user sends a message:
   - admin should receive it live without manual refetch
   - unread indicators should update live
   - the chat should append the new message without rerendering the whole thread
   - no scroll jump
   - no flicker

4. When admin sends a mentor reply:
   - user should receive it live immediately
   - unread indicators should update live even if the user is currently on another section such as profile/subscription
   - user should hear a subtle notification sound only for newly received mentor messages
   - badge/count near the chat menu item should update live
   - if user is currently inside the active open conversation, append the message instantly without full redraw

5. Notification behavior:
   - add a lightweight “ding” sound for newly received messages only
   - do NOT play sound for messages sent by the current viewer
   - do NOT repeatedly play sound on reconnect or initial load
   - only play sound for genuinely new inbound messages

6. Preserve read/unread logic:
   - do not break existing unread model
   - when a message is seen/opened in the relevant conversation view, unread state should clear correctly
   - live counts should decrease/clear when messages become read

7. Preserve current message list rendering style as much as possible.
   Only update the minimal changed parts:
   - append incoming messages
   - update status/badge states
   - avoid full component remounts
   - avoid refetching the full page

8. Admin page must no longer rely on a manual “Refetch” workflow for real-time conversation updates.
   If the refetch button exists, it can remain as a fallback utility, but the conversation itself must stay live automatically.
